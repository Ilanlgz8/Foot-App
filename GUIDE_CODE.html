Introduction

C'est quoi StatFootix ?

StatFootix est une application web React qui affiche les matchs de foot en temps réel — scores, minutes live, classements et actualités. Elle combine plusieurs APIs pour donner les infos les plus précises et les plus rapides possible.

📄 4 sections

Accueil live, Matchs à venir, Résultats, Classement

🔌 4 APIs

football-data.org, API-Football, ESPN, GNews

⏱️ Minutes précises

Calculées à la seconde près via des timestamps localStorage

☁️ Netlify

Fonctions serverless qui cachent les clés API secrètes

💡 Niveau requis : Ce guide part du principe que tu connais les bases JS (fonctions, tableaux, objets). Tout le reste — React, hooks, localStorage, polling — est expliqué ici.

L'architecture globale

Le navigateur ne touche jamais directement les APIs externes. Il passe toujours par une Netlify Function (un mini-serveur) qui ajoute la clé API secrète avant de transmettre la requête.

🔐 Pourquoi Netlify Functions ? Si tu mettais la clé API directement dans le code React, n'importe qui pourrait l'ouvrir dans les DevTools du navigateur et la voler. Les Netlify Functions tournent côté serveur — la clé n'est jamais envoyée au navigateur.

Bases React

React Query — charger et cacher les données

React Query est une bibliothèque qui gère automatiquement la récupération de données depuis une API. Sans elle, il faudrait écrire soi-même la gestion du chargement, des erreurs, du cache, des re-fetches automatiques… C'est beaucoup de code répétitif.

Le pattern de base

import { useQuery } from '@tanstack/react-query'

function useStandings(competition) {
  const { data, isLoading, error } = useQuery({

    // 1. La clé : identifie cette requête dans le cache global
    //    Si la clé change (autre compétition) → React Query refetch automatiquement
    queryKey: ['standings', competition],

    // 2. La fonction qui va chercher les données sur le réseau
    queryFn: async () => {
      const res  = await fetch(`/api/v4/competitions/${competition}/standings`)
      const json = await res.json()
      return json.standings
    },

    // 3. Options :
    staleTime:       5 * 60 * 1000,  // données "fraîches" pendant 5min (pas de refetch inutile)
    refetchInterval: 60_000,          // refetch automatique toutes les 60s
  })

  return { standings: data ?? [], loading: isLoading }
}

queryKey

L'identifiant du cache. Même clé dans deux composants = un seul fetch partagé.

staleTime

Durée où les données sont "fraîches". Passé ce délai, React Query refetch en arrière-plan.

refetchInterval

Pour les données live : refetch automatique toutes les N ms. 30_000 = 30 secondes.

enabled

Si false, la requête ne s'exécute pas. Utile pour attendre une condition.

💡 Cache partagé automatique : Si LiveWidget et Accueil appellent tous les deux useLiveMatches() qui a la même queryKey: ['liveMatches'], React Query ne fait qu'une seule requête réseau et distribue les données aux deux composants. C'est gratuit, pas besoin de le configurer.

Les hooks simples

Un hook React c'est une fonction qui commence par use. Elle peut contenir de la logique réutilisable (requêtes, états, effets) qu'on appelle dans les composants.

useNews — les actualités

Charge les articles foot depuis GNews, filtre ceux qui parlent vraiment de foot (pas juste de sport en général), et élimine les doublons par titre.

const FOOTBALL_KEYWORDS = ['football', 'ligue', 'champions', 'transfert', 'ballon d'or', ...]

export function useNews() {
  const { data, isLoading } = useQuery({
    queryKey: ['news'],
    queryFn: async () => {
      const res  = await fetch('/news?q=football&lang=fr')
      const json = await res.json()

      // Map pour dédupliquer : clé = titre, valeur = article
      const byTitle = new Map()
      for (const article of json.articles) {
        byTitle.set(article.title, article)
      }

      // Ne garder que les articles qui mentionnent un mot-clé foot
      return [...byTitle.values()]
        .filter(a => FOOTBALL_KEYWORDS.some(kw => a.title.toLowerCase().includes(kw)))
        .sort((a, b) => new Date(b.publishedAt) - new Date(a.publishedAt))
        .slice(0, 12)  // top 12 articles seulement
    },
    staleTime: 30 * 60 * 1000,  // cache 30min (les actus changent pas toutes les secondes)
  })
  return { news: data ?? [], isLoading }
}

useTodayMatches — les matchs du jour

Charge les matchs d'une date choisie. Football-data.org ne retourne pas la Coupe du Monde dans l'endpoint global (limite du plan gratuit), donc on fait 2 requêtes séparées qu'on fusionne.

async function fetchForDate(date) {
  // Requête 1 : toutes les ligues européennes + Ligue des Champions
  const euroMatches = await safeFetch(
    `/api/v4/matches?dateFrom=${date}&dateTo=${date}&competitions=CL,PL,FL1,PD,BL1,SA`
  )

  await delay(700)  // 700ms de pause pour respecter la limite 10 req/min

  // Requête 2 : Coupe du Monde séparément
  const wcMatches = await safeFetch(
    `/api/v4/competitions/WC/matches?dateFrom=${date}&dateTo=${date}`
  )

  // Fusionner + trier par heure de coup d'envoi
  return [...euroMatches, ...wcMatches]
    .sort((a, b) => new Date(a.utcDate) - new Date(b.utcDate))
}
♻️ Cache intelligent : Si des matchs sont en cours → cache 2min. Sinon → cache 1h. Pour d'autres jours → cache 6h. Le hook précharge aussi J-1 et J+1 en arrière-plan pour que la navigation entre les jours soit instantanée.

useLiveMatches — les matchs en cours

Ce hook récupère tous les matchs actuellement en direct, peu importe la date affichée dans le panneau principal. C'est le hook "global" qui alimente le widget Live.

export function useLiveMatches() {
  const { data } = useQuery({
    queryKey: ['liveMatches'],  // clé partagée → 1 seul fetch pour tout le site
    queryFn: async () => {
      // Deux statuts différents dans football-data.org :
      // IN_PLAY = match en cours, PAUSED = mi-temps
      const [r1, r2] = await Promise.all([
        fetch('/api/v4/matches?status=IN_PLAY'),
        fetch('/api/v4/matches?status=PAUSED'),
      ])
      const live   = r1.ok ? (await r1.json()).matches ?? [] : []
      const paused = r2.ok ? (await r2.json()).matches ?? [] : []

      // trackMatchState → enregistre les transitions dans localStorage
      ;[...live, ...paused].forEach(trackMatchState)

      // Dédupliquer (un match ne peut pas être IN_PLAY et PAUSED à la fois)
      const seen = new Set()
      return [...live, ...paused].filter(m => {
        if (seen.has(m.id)) return false
        seen.add(m.id)
        return true
      })
    },
    // Refetch adaptatif : si des matchs sont en cours → toutes les 30s
    // sinon → toutes les 5min (économise les requêtes quand rien se passe)
    refetchInterval: (query) => {
      const hasLive = (query.state.data ?? []).length > 0
      return hasLive ? 30_000 : 5 * 60_000
    },
    staleTime: 20_000,
  })
  return data ?? []
}
💡 Pourquoi deux requêtes ? Football-data.org sépare les statuts. IN_PLAY = match en cours. PAUSED = mi-temps. Ce sont deux endpoints différents, on les combine pour avoir une liste complète de "tout ce qui se joue maintenant".

useEspnScores — les scores ultra-rapides

Football-data.org met les scores à jour avec ~1 minute de délai. ESPN (API non-officielle, gratuite) les met à jour en moins de 10 secondes. C'est la source principale pour les scores live.

Le problème : les noms sont différents

ESPN et football-data.org n'utilisent pas les mêmes codes de compétition ni les mêmes noms d'équipes. Il faut donc faire un mapping et un matching par nom.

// Les codes de compétition football-data → slugs ESPN
const ESPN_SLUG = {
  FL1: 'fra.1',       // Ligue 1
  PL:  'eng.1',       // Premier League
  PD:  'esp.1',       // La Liga
  SA:  'ita.1',       // Serie A
  CL:  'uefa.champions',
  WC:  'fifa.world',
}

// Matcher un événement ESPN avec notre match football-data par noms d'équipe
function matchEvent(espnEvent, fdMatches) {
  const espnHome = normalize(espnEvent.home.team.displayName)  // ex: "france"
  const espnAway = normalize(espnEvent.away.team.displayName)

  return fdMatches.find(m => {
    const h = normalize(m.homeTeam.name)  // ex: "france nationale"
    const a = normalize(m.awayTeam.name)
    // Compare les 5 premiers caractères pour absorber les variantes de noms
    return h.startsWith(espnHome.slice(0, 5)) && a.startsWith(espnAway.slice(0, 5))
  })
}

// Normaliser : enlever les accents, minuscules
function normalize(s = '') {
  return s.toLowerCase()
    .replace(/[àáâ]/g, 'a')
    .replace(/[éèê]/g, 'e')
    .trim()
}

Ce que le hook retourne

Un objet { matchId: { home: 2, away: 1 } } qu'on lit dans les MatchCards :

// Dans un composant MatchCard :
const score = espnScores[match.id]

// ESPN d'abord (< 10s), football-data.org en fallback (~1min)
const homeGoals = score?.home ?? match.score?.fullTime?.home
const awayGoals = score?.away ?? match.score?.fullTime?.away

matchStateTracker — la mémoire des matchs

Ce fichier gère une mini-base de données dans le localStorage du navigateur. Pour chaque match en direct, il mémorise des timestamps (des moments précis) qui permettent ensuite de calculer la minute affichée.

💡 localStorage c'est un espace de stockage dans le navigateur qui persiste même après rechargement. C'est différent des variables React qui se remettent à zéro quand tu recharges la page. On s'en sert pour garder les timestamps des KO, MT, et reprises.

Les 3 timestamps mémorisés

kickoffAt

Timestamp Unix exact du vrai coup d'envoi. Peut différer de l'heure prévue si le match commence en retard.

pausedAt

Timestamp exact quand la mi-temps a commencé. Déclenche l'affichage "MT" et active la fenêtre 2.

half2Start

Timestamp exact quand la 2ème mi-temps a repris. Clé pour calculer les minutes 46'→90'+.

// Clé localStorage : "foot_ms_123456" (foot_ms_ + l'id du match)
const key = (id) => `foot_ms_${id}`

// Retourner l'état mémorisé d'un match
export function getMatchState(matchId) {
  return JSON.parse(localStorage.getItem(key(matchId)) || '{}')
  // → { kickoffAt: 1749..., pausedAt: 1749..., half2Start: 1749... }
}

// Enregistrer les transitions d'état (appelé quand football-data.org répond)
export function trackMatchState(match) {
  if (!match?.id) return
  const stored = getMatchState(match.id)

  // Première fois qu'on voit PAUSED → noter l'heure exacte de la MT
  if (match.status === 'PAUSED' && !stored.pausedAt) {
    stored.pausedAt = Date.now()
    localStorage.setItem(key(match.id), JSON.stringify(stored))
  }

  // IN_PLAY après avoir vu PAUSED → c'est la reprise de la 2ème MT
  if (match.status === 'IN_PLAY' && stored.pausedAt && !stored.half2Start) {
    stored.half2Start = Date.now()
    localStorage.setItem(key(match.id), JSON.stringify(stored))
  }
}

// Effacer TOUS les états (pour la recalibration manuelle)
export function clearAllMatchStates() {
  Object.keys(localStorage)
    .filter(k => k.startsWith('foot_ms_'))
    .forEach(k => localStorage.removeItem(k))
}

La liste des matchs "suivis"

Une autre clé localStorage (foot_tracked_matches) mémorise quels matchs l'utilisateur a activé pour le polling API-Football. Seuls ces matchs consomment du quota.

// Récupérer les IDs suivis (retourne un Set pour les .has() rapides)
export function getTrackedMatches() {
  return new Set(JSON.parse(localStorage.getItem('foot_tracked_matches') || '[]'))
}

// Ajouter ou retirer un match du suivi (toggle)
export function toggleTrackedMatch(matchId) {
  const ids = getTrackedMatches()
  if (ids.has(String(matchId))) ids.delete(String(matchId))
  else ids.add(String(matchId))
  localStorage.setItem('foot_tracked_matches', JSON.stringify([...ids]))
}

calcMinute — afficher la bonne minute

Fonction pure qui calcule ce qu'il faut afficher à la place de la minute (ex: 73', MT, 90+2'). Elle utilise une hiérarchie de précision : elle prend les données les plus précises disponibles.

Ordre de priorité :
half2Start connu
→
Calcul 2ème MT à la seconde (46'→90'+)
pausedAt connu, pas half2Start
→
Affiche "MT"
kickoffAt connu
→
Calcul 1ère MT précis (1'→45'+)
Rien de connu
→
Heuristique depuis utcDate (approximation)
export function calcMinute(match) {
  // Statut PAUSED de football-data.org → toujours "MT" directement
  if (match.status === 'PAUSED') return 'MT'
  if (match.status !== 'IN_PLAY') return null

  const state = getMatchState(match.id)

  // ── CAS 1 : on connaît le début de la 2ème MT ──
  if (state.half2Start) {
    // Combien de minutes se sont écoulées depuis la reprise ?
    const min2 = Math.floor((Date.now() - state.half2Start) / 60_000) + 1
    if (min2 <= 45) return `${45 + min2}'`   // 46' ... 90'
    return `90+${min2 - 45}'`                // 90+1' ... 90+5' etc.
  }

  // ── CAS 2 : MT confirmée mais reprise pas encore détectée par API-Football ──
  if (state.pausedAt) return 'MT'

  // ── CAS 3 : on connaît le vrai coup d'envoi ──
  if (state.kickoffAt) {
    const min1 = Math.floor((Date.now() - state.kickoffAt) / 60_000)
    if (min1 <= 45) return `${Math.max(1, min1)}'`
    if (min1 - 45 <= 8) return `45+${min1 - 45}'`
  }

  // ── CAS 4 : heuristique (aucun timestamp connu) ──
  // Approximation : suppose que la 2H commence 64min après l'heure prévue
  // (45min de 1H + ~4min de temps additionnel + ~15min de pause MT)
  const elapsed = Math.floor((Date.now() - new Date(match.utcDate)) / 60_000)
  if (elapsed <= 45) return `${Math.max(1, elapsed)}'`
  if (elapsed <= 64) return 'MT'
  const half2 = elapsed - 64
  if (half2 <= 45) return `${45 + half2}'`
  return `90+${half2 - 45}'`
}
⚠️ L'heuristique est imprécise. Si le match commence en retard ou si la pause dure 20 minutes au lieu de 15, elle peut afficher la mauvaise minute. C'est pour ça qu'on utilise API-Football pour enregistrer les vrais timestamps — les cas 1, 2, 3 sont toujours plus précis.

useLiveMinute — le chef d'orchestre

Le hook le plus complexe du projet. Il décide quand appeler API-Football (pour économiser le quota de 100 req/jour), et met à jour le localStorage avec les timestamps précis quand il reçoit les données.

🎯 Le principe : au lieu de poller API-Football toutes les minutes pendant 90min (= 90 requêtes/match), on ne poll que pendant 4 "fenêtres" précises. Un match entier consomme ~10-20 requêtes au lieu de 90+. Et une seule requête /apifootball?live=all couvre TOUS les matchs en même temps. 
export function useLiveMinute(matches) {
  const queryClient  = useQueryClient()
  const matchesRef   = useRef(matches)
  matchesRef.current = matches  // toujours la liste la plus à jour sans redémarrer l'intervalle

  useEffect(() => {
    const tick = async () => {
      const trackedIds = getTrackedMatches()

      // Est-ce qu'au moins un match nécessite un poll maintenant ?
      const needsPoll = matchesRef.current.some(m => isInPollingWindow(m, trackedIds))

      if (needsPoll) await pollApiFootball(matchesRef.current, queryClient)
    }

    tick()  // poll immédiatement au démarrage
    const id = setInterval(tick, 60_000)  // puis toutes les 60 secondes
    return () => clearInterval(id)        // nettoyage quand le composant est démonté
  }, [queryClient])

  // Recalibration manuelle : vide les états + poll forcé (ignore les fenêtres)
  // Utilise intentionnellement 1 requête API-Football
  const recalibrate = useRef(async () => {
    clearAllMatchStates()
    await pollApiFootball(matchesRef.current, queryClient)
  })

  return { recalibrate: recalibrate.current }
}

Les 4 fenêtres de polling

La fonction isInPollingWindow(match, trackedIds) décide si ce match doit déclencher un poll API-Football maintenant. Elle vérifie 4 conditions :

W0
1ère MT
MT
W1→W2
2ème MT
W3 à ~90'
FT
Heure prévue45'~60'~75'90'Fin
FENÊTRE 0
Coup d'envoi
🚀 Détection du KO Active dès l'heure prévue du match. S'applique à tous les matchs (pas besoin d'être suivi). Surveille le passage de TIMED → IN_PLAY et enregistre kickoffAt depuis l'elapsed API-Football.
FENÊTRE 1
Mi-temps
⏸️ Détection de la MT S'active entre la 45ème et 60ème minute. S'active aussi si football-data.org affiche déjà PAUSED (fix race condition). Quand API-Football répond HT, on enregistre pausedAt.
FENÊTRE 2
Reprise 2H
▶️ Détection de la reprise S'active 15 minutes après pausedAt. Quand API-Football répond 2H, on reconstruit half2Start depuis l'elapsed. Seulement si elapsed < 90 (API-Football cap à 90 en temps additionnel).
FENÊTRE 3
Fin de match
🏁 Détection du FT S'active quand la 2ème MT dure depuis 44+ minutes. Surveille FT (fin), AET (prolongations), PEN (tirs au but). Nettoie le localStorage.

La reconstruction des timestamps

Le point le plus subtil du système : quand API-Football dit "elapsed = 52 minutes", on sait que la 2H a commencé il y a 52-46 = 6 minutes. Donc on peut reconstruire l'heure exacte du début de la 2H :

// API-Football retourne : { status: { short: '2H', elapsed: 52 } }
// La 2H commence toujours à la 46ème minute (convention)
// Donc elapsed=52 signifie "6 minutes de 2H écoulées"

const half2Start = Date.now() - (apiElapsed - 46) * 60_000
//                  Maintenant   -   6 minutes   = timestamp du début de la 2H

// Puis calcMinute utilise ce timestamp :
const min2 = Math.floor((Date.now() - half2Start) / 60_000) + 1
// → 6 + 1 = 7 → affiche 45+7 = 52'  ✓

// ⚠️ Piège : API-Football cap elapsed à 90 pendant le temps additionnel
// Si elapsed >= 90 et qu'on recalcule half2Start → min2 resterait bloqué à 45 → 90' figé
// Solution : seulement recalculer si apiElapsed < 90
if (apiElapsed != null && apiElapsed > 45 && apiElapsed < 90) {
  setHalf2Start(match.id, half2Start)
}
// Après 90', half2Start reste figé à sa dernière valeur → calcMinute continue d'avancer normalement

Accueil.jsx — le composant principal

C'est le composant central de la page d'accueil. Il rassemble toutes les données et les distribue aux sous-composants.

function Accueil() {
  // ── Navigation entre les jours ──
  const [dayOffset, setDayOffset] = useState(0) // 0=aujourd'hui, 1=demain, -1=hier...

  // ── Données ──
  const { matches } = useTodayMatches(targetDate)   // matchs du JOUR AFFICHÉ
  const globalLive   = useLiveMatches()             // matchs live TOUS JOURS CONFONDUS

  // ── Système live ──
  // On fusionne les deux listes pour que useLiveMinute poll les matchs live même si
  // le joueur a navigué vers un autre jour (les matchs live ne seraient plus dans matches)
  const { recalibrate } = useLiveMinute([
    ...matches,
    ...globalLive.filter(g => !matches.some(m => m.id === g.id))
  ])

  // ESPN utilise globalLive (pas matches filtré par jour)
  const espnScores = useEspnScores(globalLive)

  // ── Suivi des matchs ──
  const [trackedIds, setTrackedIds] = useState(() => getTrackedMatches())

  // Auto-tracker si 5 matchs ou moins ce jour (quota suffisant, pas besoin de choisir)
  useEffect(() => {
    if (matches.length > 0 && matches.length <= 5) {
      matches.filter(m => m.status !== 'FINISHED').forEach(m => toggleTrackedMatch(m.id))
      setTrackedIds(getTrackedMatches())
    }
  }, [matches.length])

  // ── Ticker 30s : force un re-render pour que calcMinute avance ──
  const [, setTick] = useState(0)
  useEffect(() => {
    const id = setInterval(() => setTick(t => t + 1), 30_000)
    return () => clearInterval(id)
  }, [])

  return (
    <section>
      <LiveWidget espnScores={espnScores} trackedIds={trackedIds} onRecalibrate={recalibrate} />
      <MatchPanel matches={matches} espnScores={espnScores} />
    </section>
  )
}
🔁 Pourquoi le ticker à 30s ? calcMinute utilise Date.now() pour calculer la minute. Mais React ne re-render pas automatiquement à chaque seconde — il faut lui dire de le faire. On utilise un faux state qui change toutes les 30s pour forcer le re-render et faire avancer l'horloge dans l'interface.

LiveWidget — le widget EN DIRECT

Affiché en haut de l'accueil quand des matchs sont en cours. Indépendant du jour affiché dans MatchPanel — il montre toujours les matchs en cours en temps réel.

function LiveWidget({ espnScores, trackedIds, onRecalibrate }) {
  // useLiveMatches() est déjà appelé dans Accueil → même queryKey → 0 requête supplémentaire
  const matches = useLiveMatches()

  if (matches.length === 0) return null  // pas de matchs → widget invisible

  return (
    <div className="liveWidget">
      <div className="header">
        <span>EN DIRECT</span>
        {/* ⟳ bouton discret de recalibration des minutes */}
        {onRecalibrate && (
          <button className="recalBtn" onClick={onRecalibrate} title="Recalibrer les minutes">
            ⟳
          </button>
        )}
      </div>
      {matches.map(match => (
        <MatchCard
          match={match}
          tracked={trackedIds.has(String(match.id))}
          espnScore={espnScores[match.id]}
          // Pas de onTrack ici : le widget n'a pas de bouton "Suivre"
        />
      ))}
    </div>
  )
}
✅ Persiste quand on navigue entre les jours : même si l'utilisateur va voir les matchs de demain dans MatchPanel, le widget continue d'afficher les matchs en cours grâce à useLiveMatches() qui est global. Et les minutes continuent d'avancer car globalLive est passé à useLiveMinute dans Accueil.

MatchPanel — les matchs du jour sélectionné

Affiche la liste des matchs pour la date choisie, avec navigation J-1/J+1. Lié au jour, contrairement au LiveWidget.

Le bouton "Suivre"

Apparaît uniquement dans MatchPanel et seulement si :

  • Il y a plus de 5 matchs ce jour (sinon ils sont tous auto-trackés)
  • Le match n'est pas terminé
  • Moins de 5 matchs sont déjà suivis OU ce match est déjà suivi (pour pouvoir le désactiver)
// Logique d'affichage du bouton Suivre

// netlify/functions/apifootball.js
export default async function (req, ctx) {
  const res = await fetch('https://v3.football.api-sports.io/fixtures?live=all', {
    headers: {
      // process.env.APIFOOTBALL_KEY = variable définie dans Netlify Dashboard
      // N'est JAMAIS envoyée au navigateur — reste côté serveur
      'x-rapidapi-key': process.env.APIFOOTBALL_KEY,
      'x-rapidapi-host': 'v3.football.api-sports.io',
    }
  })

  // On transfère le header de quota au navigateur pour surveiller combien il reste
  const remaining = res.headers.get('x-ratelimit-requests-remaining')
  const headers   = { 'Content-Type': 'application/json' }
  if (remaining) headers['x-quota-remaining'] = remaining

  return new Response(await res.text(), { headers })
}

Cache local — localCache.js

En complément du cache React Query, on a un cache maison dans localStorage pour les matchs. Avantage : le cache survit aux rechargements de page, ce qui permet d'afficher les données immédiatement sans attendre la requête réseau.

// Écrire des données dans le cache avec une durée de vie
export function writeCache(key, data, ttl) {
  localStorage.setItem(key, JSON.stringify({
    data,
    savedAt:   Date.now(),
    expiresAt: Date.now() + ttl,  // ex: Date.now() + 2*60*1000 = expire dans 2min
  }))
}

// Lire le cache — retourne null si expiré
export function readCache(key) {
  const raw = JSON.parse(localStorage.getItem(key))
  if (!raw || Date.now() > raw.expiresAt) return null  // expiré → null
  return raw.data
}

// Lire même si expiré ("stale") — utilisé comme initialData dans React Query
// Permet d'afficher des données datées pendant que le refetch s'effectue
export function readCacheStale(key) {
  return JSON.parse(localStorage.getItem(key))?.data ?? null
}
💡 Stale-while-revalidate : quand tu ouvres la page, React Query utilise d'abord les données du cache local (même expirées) comme initialData pour afficher quelque chose immédiatement, puis refetch en arrière-plan. Résultat : jamais d'écran de chargement blanc si tu as déjà visité la page récemment.
