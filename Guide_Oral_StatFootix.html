1

Présentation du projet

~1 min ▼
Ce que tu montres
Titre, stack technique — React, TanStack Query, Netlify Functions, football-data.org
Ce que tu dis

StatFootix c'est une appli web de stats football en temps réel. L'idée : suivre les matchs en direct, consulter classements, résultats et buteurs — dans une interface propre.

Côté stack : React 18 pour l'UI, TanStack Query pour toute la gestion des données et du cache, et Netlify Functions comme couche serveur pour sécuriser les appels aux APIs externes.

2

Architecture — comment les données circulent

~2 min ▼
Ce que tu montres
Schéma 3 couches : Navigateur → Netlify Function → APIs externes
Ce que tu dis

L'architecture c'est trois couches. Le navigateur ne parle jamais directement aux APIs externes — il passe toujours par une Netlify Function qui joue le rôle de proxy sécurisé.

On a 4 sources de données : football-data.org pour les matchs et classements, api-football.com pour les minutes précises en direct, ESPN pour les scores quasi-instantanés, et 4 flux RSS agrégés pour les actus foot.

Le navigateur appelle /api/... ou /espn/... — il ne voit jamais la vraie URL ni la clé API.

3

Sécuriser la clé API avec Netlify Functions

~2 min ▼
Ce que tu montres
Comparaison ❌ sans proxy (clé visible DevTools) vs ✅ avec Netlify Function
Ce que tu dis

Le problème de base : si tu mets ta clé API directement dans le code React, elle est visible dans les DevTools du navigateur. N'importe qui peut la récupérer et consommer ton quota.

La solution : une Netlify Function côté serveur. React appelle /api/v4/matches, Netlify intercepte, ajoute la vraie clé depuis une variable d'environnement, et retransmet la réponse. La clé ne quitte jamais le serveur.

Détail important : si l'API répond 429 (quota dépassé), on retransmet ce code tel quel. TanStack le détecte et conserve les données précédentes — pas de spinner, pas de page blanche.

4

TanStack Query — le pattern commun

~2 min ▼
Ce que tu montres
Code de useMatches avec annotations queryKey / queryFn / staleTime
Ce que tu dis

Toute la gestion des données suit le même pattern : un hook, une queryKey, une queryFn.

La queryKey c'est l'identifiant de la donnée en cache. Si je change de compétition, la clé change → TanStack fait un nouveau fetch automatiquement. Si deux composants utilisent la même clé en même temps — la page Résultats et le panel Accueil par exemple — il n'y a qu'une seule requête réseau. C'est le cache partagé.

Le staleTime : tant que la donnée est fraîche, aucun refetch inutile. Ça préserve le quota API.

5

Récupérer les matchs du jour

~2 min ▼
Ce que tu montres
fetchTodayMatches — 2 requêtes séparées + delay(700) + déduplication
Ce que tu dis

Premier problème découvert : l'endpoint global de football-data.org en tier gratuit ne retourne pas les matchs de la Coupe du Monde. Donc pour avoir tous les matchs d'une journée, il faut deux requêtes séparées — une pour les ligues européennes, une pour la WC.

Le delay(700) entre les deux c'est essentiel : l'API limite à 10 requêtes par minute. Sans ce délai, on prend un 429 au chargement.

Ensuite on déduplique par ID et on trie chronologiquement.

6

Afficher un match — MatchCard

~2 min ▼
Ce que tu montres
Code MatchCard — blasons, scores ESPN vs FDO, calcMinute()
Ce que tu dis

MatchCard c'est le composant de base, il affiche un match dans n'importe quel contexte : panel du jour, widget live, résultats.

Pour le score : on priorise ESPN plutôt que football-data.org, parce qu'ESPN met à jour en moins de 10 secondes alors que FDO a un délai d'environ 1 minute. Si ESPN n'a pas le score, l'opérateur ?. fait le fallback automatiquement.

Pour la minute : calcMinute() calcule côté client depuis des timestamps stockés en localStorage. Mise à jour toutes les 30s via un setInterval — sans aucun refetch réseau. Elle affiche 73', MT, 90+2' selon l'état du match.

7

Résultats — fetch + affichage

~2 min ▼
Ce que tu montres
useMatches avec queryKey partagée, affichage score + blasons + gagnant
Ce que tu dis

Le point clé ici c'est le cache partagé. La page Résultats et le panel "Résultats récents" de l'Accueil utilisent exactement le même hook avec la même queryKey. Donc une seule requête pour les deux — dès que la page Résultats se met à jour, le panel Accueil est à jour instantanément.

Pour l'affichage : les blasons viennent directement de l'URL fournie par l'API. Si l'URL est vide on affiche la première lettre du nom — fallback propre. Le gagnant est mis en évidence avec une classe CSS.

8

Classements — useStandings

~2 min ▼
Ce que tu montres
useStandings — détection automatique multi-groupes vs tableau unique
Ce que tu dis

Le classement a une complexité particulière : l'API retourne une structure différente selon la compétition. Championnat classique → un seul tableau. Coupe du Monde ou Ligue des Champions → plusieurs groupes, A, B, C...

Le hook détecte ça automatiquement : si realGroups.length > 1, on est en multi-groupes, sinon tableau unique. Le composant n'a pas besoin de savoir quelle compétition est chargée — il reçoit groups et standings et s'adapte tout seul.

9

Top buteurs — useScorers

~1 min 30 ▼
Ce que tu montres
useScorers — enabled, staleTime 10min, affichage top 3 stylisé
Ce que tu dis

Détail intéressant : enabled: !!compId. TanStack permet de désactiver une requête conditionnellement — si aucune compétition n'est sélectionnée, la requête ne part pas du tout. Pas besoin d'un if dans le composant, TanStack gère ça nativement.

Le staleTime est à 10 minutes parce que les stats de buteurs ne changent qu'après un but — pas besoin de rafraîchir toutes les 30 secondes. Ça préserve le quota.

10

Scores en direct — overlay ESPN

~2 min ▼
Ce que tu montres
useEspnScores — fuzzy matching, refetchInterval 10s, enabled conditionnel
Ce que tu dis

Le vrai problème du temps réel : football-data.org free tier met environ 1 minute à mettre à jour les scores. Pour un match en direct c'est trop long.

Solution : ESPN a une API publique non officielle qui met à jour en moins de 10 secondes. On la poll toutes les 10s uniquement quand un match est en cours — enabled s'en charge. Zéro requête si rien n'est live.

Le challenge c'est le fuzzy matching : les noms d'équipes diffèrent entre ESPN et FDO. "PSG" vs "Paris Saint-Germain". Solution : on normalise (minuscules, accents retirés) et on compare les 5 premiers caractères. "paris" correspond à "paris saint-germain". Ça marche pour 99% des cas.

11

Conclusion

~1 min 30 ▼
Ce que tu montres
4 points clés : proxy Netlify, pattern useQuery, ESPN overlay, enabled + staleTime
Ce que tu dis

Ce que j'ai retenu de ce projet :

Le proxy Netlify c'est non-négociable dès qu'il y a une clé API — une ligne dans netlify.toml et la clé ne quitte plus le serveur.

TanStack Query simplifie énormément : un seul pattern couvre fetch, cache, loading, error et refetch automatique.

ESPN en overlay c'est une solution pragmatique quand l'API principale est trop lente — on garde FDO comme source de vérité et ESPN pour la fraîcheur.

enabled + staleTime = zéro requête inutile, quota préservé.

Questions pièges — réponses prêtes

Pourquoi pas un backend custom (Node/Express) à la place de Netlify Functions ? ▼
Netlify Functions c'est suffisant pour ce besoin — zéro infrastructure à gérer, déploiement automatique avec le front, pas de serveur à maintenir. Pour scaler, on passerait sur un backend dédié, mais là c'est du projet perso.
Pourquoi TanStack Query plutôt que SWR ou Redux ? ▼
TanStack Query est le standard React pour la gestion des données serveur. Plus complet que SWR (invalidation, préchargement, cache partagé natif), et Redux c'est pour le state UI — pas pour du fetching. C'est le bon outil pour le bon besoin.
C'est quoi les limites de l'appli ? ▼
Le free tier football-data.org limite à 10 req/min et 100 req/jour pour api-football.com. L'app est pensée pour un usage personnel — pour plusieurs utilisateurs simultanés il faudrait une couche de cache côté serveur (Redis par exemple) pour ne pas multiplier les appels.
ESPN c'est légal d'utiliser leur API non officielle ? ▼
C'est une API publique non documentée, utilisée par des milliers de projets open source. Dans le cadre d'un projet perso sans monétisation, c'est toléré. Pour un projet commercial il faudrait une licence officielle.
Pourquoi localStorage pour les timestamps et pas le state React ? ▼
Les timestamps de kickoff et mi-temps doivent survivre aux re-renders et aux refreshs de page. Si l'utilisateur actualise en cours de match, le localStorage permet de retrouver exactement où on en était — le state React lui est réinitialisé.
