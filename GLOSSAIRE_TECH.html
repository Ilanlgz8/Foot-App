Hooks React natifs
useState

Crée une variable réactive dans un composant. Chaque fois que la valeur change, React re-rend le composant. Dans ce projet, utilisé pour les états locaux d'UI (page active, modal ouverte, index courant…).

src/accueil/ResultPanel.jsx · src/accueil/NewsCarousel.jsx
// ResultPanel.jsx — index du jour affiché

// NewsCarousel.jsx — page du carousel + auto-défilement
const [page, setPage] = useState(0)
useEffect(() => {
  const id = setInterval(() => {
    setPage(p => (p + 1) % pages)   // boucle circulaire
  }, 20_000)
  return () => clearInterval(id)    // cleanup au démontage
}, [pages])
useEffect

Exécute du code en réaction à un changement (montage, dépendances). La fonction retournée est un cleanup — elle s'exécute quand le composant se démonte. Dans ce projet, utilisé pour les timers, prefetch et IntersectionObserver.

src/components/Accueil.jsx
// Ticker 30s : force un re-render pour actualiser les minutes sans refetch réseau
useEffect(() => {
  const id = setInterval(() => setTick(t => t + 1), 30_000)
  return () => clearInterval(id)
}, [])

// Prefetch des jours adjacents 2s après le chargement
useEffect(() => {
  const t = setTimeout(() => {
    prefetchMatchesForDate(queryClient, tomorrow)
    prefetchMatchesForDate(queryClient, yesterday)
  }, 2000)
  return () => clearTimeout(t)
}, [dayOffset])

useMemo mémorise le résultat d'un calcul coûteux. useCallback mémorise une fonction. Les deux évitent de recalculer à chaque render si les dépendances n'ont pas changé.

Custom Hooks
Custom Hook

Une fonction qui commence par use et qui peut appeler d'autres hooks. Permet d'extraire la logique hors des composants pour la réutiliser. Les 13 hooks de ce projet sont tous des custom hooks.

src/hooks/useLiveMatches.js
// Un custom hook = une fonction useXxx qui retourne des données
export function useLiveMatches() {
  const { data } = useQuery({
    queryKey: ['liveMatches'],
    queryFn: async () => { /* ... fetch IN_PLAY + PAUSED */ },
    refetchInterval: 30_000,
  })
  return data ?? []
}

// Dans le composant, on appelle simplement :
const liveMatches = useLiveMatches()  // propre, pas de logique dans le composant

TanStack Query — useQuery
useQuery

Le hook central de TanStack. Il fetche, met en cache, et synchronise des données distantes. Remplace complètement le pattern useEffect + fetch + useState et gère automatiquement loading, error, et refetch.

src/hooks/useLiveMatches.js
const { data, isLoading, error } = useQuery({
  queryKey:  ['liveMatches'],                 // identifiant du cache
  queryFn:   async () => { /* fetch */ },      // fonction qui fetche les données
  staleTime: 20_000,                           // fraîches 20s → pas de refetch
  refetchInterval: 30_000,                     // re-fetch auto toutes les 30s
  retry: false,                                 // pas de retry sur erreur
})
Avantage clé : Si deux composants appellent useQuery avec le même queryKey, TanStack ne fait qu'une seule requête réseau et partage le résultat. C'est pour ça que LiveWidget et Accueil.jsx partagent les données live sans double fetch.
TanStack Query — queryKey
queryKey

L'identifiant unique du cache. C'est un tableau. Si une valeur dans le tableau change, TanStack considère que c'est une nouvelle requête et refetch. Si deux useQuery partagent le même queryKey → même cache, une seule requête.

src/hooks/useTodayMatches.js
// queryKey contient la date → chaque jour a son propre cache
useQuery({
  queryKey: ['todayMatches', date],   // ['todayMatches', '2026-06-16']
  queryFn: () => fetchTodayMatches(date),
})
// Changer date → nouveau queryKey → refetch automatique
// Revenir à une date → queryKey déjà en cache → données immédiates

// Autres exemples dans le projet :
queryKey: ['liveMatches']              // useLiveMatches
queryKey: ['news']                     // useNews
queryKey: ['standings', compId]        // useStandings
queryKey: ['matchDetails', matchId]    // useMatchDetails
TanStack Query — staleTime & gcTime
staleTime

Durée pendant laquelle les données sont considérées "fraîches". Pendant ce délai, TanStack ne refetch pas même si tu remountes le composant ou changes d'onglet. Après ce délai, les données sont "périmées" (stale) mais toujours affichées en attendant le refetch.

src/hooks/ — plusieurs fichiers
// useLiveMatches — données live très courtes durée
staleTime: 20_000           // fraîches 20s

// useNews — actus peuvent attendre 30min
staleTime: 30 * 60 * 1000   // fraîches 30min

// useMatchDetails — détails d'un match terminé très stables
staleTime: 60 * 60 * 1000   // fraîches 1h

// useScorers — stats changent après chaque journée
staleTime: 10 * 60 * 1000   // fraîches 10min
gcTime

Durée pendant laquelle les données restent en mémoire après que le composant se démonte. Après ce délai, le garbage collector supprime l'entrée du cache. Dans ce projet, mis à 24h pour que les données survivent à la navigation entre les onglets.

src/main.jsx
// Configuration globale dans main.jsx
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      gcTime: 1000 * 60 * 60 * 24,  // garde 24h en mémoire (et localStorage)
      retry: false,
      refetchOnWindowFocus: false,    // pas de refetch au retour sur l'onglet
    }
  }
})
TanStack Query — refetchInterval
refetchInterval

Lance un refetch automatique à intervalle régulier (polling). Peut être une valeur fixe ou une fonction qui reçoit le state actuel du cache pour décider dynamiquement de l'intervalle.

src/hooks/useLiveMatches.js
// Valeur fixe
refetchInterval: 30_000   // toutes les 30s

// Fonction dynamique : adapte l'intervalle selon les données actuelles
refetchInterval: (query) => {
  const hasLive = (query.state.data ?? []).length > 0
  return hasLive ? 30_000 : 5 * 60_000
  // 30s si matchs en cours, 5min sinon → économise le quota API
}

// useTodayMatches — encore plus adaptatif
refetchInterval: (query) => {
  const hasLive = matches.some(m =>
    m.status === 'IN_PLAY' || m.status === 'PAUSED'
  )
  return hasLive ? 2 * 60_000 : 10 * 60_000
}
TanStack Query — initialData
initialData

Données affichées immédiatement au premier render, avant même que la requête parte. Élimine le spinner de chargement si on a des données en cache local. TanStack refetch en arrière-plan si elles sont périmées.

src/hooks/useTodayMatches.js
export function useTodayMatches(date) {
  const cacheKey = `matches_${date}`

  return useQuery({
    queryKey: ['todayMatches', date],
    queryFn: () => fetchTodayMatches(date),

    // Données immédiates depuis localStorage → zéro spinner
    initialData: () => readCacheStale(cacheKey) ?? undefined,

    // TanStack a besoin de savoir QUAND ces données ont été sauvées
    // pour décider si elles sont encore fraîches
    initialDataUpdatedAt: () => getCacheSavedAt(cacheKey),
  })
}
TanStack Query — enabled
enabled

Contrôle si la requête doit se lancer. enabled: false = la requête est suspendue, aucun fetch ne part. Utile pour des requêtes conditionnelles (modal fermée, id manquant…).

src/hooks/useMatchDetails.js · src/hooks/useEspnScores.js
// useMatchDetails — ne fetche pas si la modal est fermée (matchId = null)
useQuery({
  queryKey: ['matchDetails', matchId],
  queryFn: () => fetch(`/api/v4/matches/${matchId}`),
  enabled: !!matchId,   // null → false → pas de requête
})

// useEspnScores — ne fetche pas si aucun match live
useQuery({
  queryKey: ['espnScores'],
  queryFn: () => fetchEspnScores(liveMatches),
  enabled: liveMatches.length > 0,   // économise des requêtes ESPN inutiles
  refetchInterval: 10_000,
})
TanStack Query — Persistance localStorage
PersistQueryClientProvider

Sauvegarde tout le cache TanStack dans localStorage automatiquement. Quand l'utilisateur revient sur l'app (même après avoir fermé l'onglet), les données s'affichent instantanément. TanStack refetch silencieusement si les données sont périmées.

src/main.jsx
import { PersistQueryClientProvider } from '@tanstack/react-query-persist-client'
import { createSyncStoragePersister } from '@tanstack/query-sync-storage-persister'

// 1. Créer le persister — lie TanStack au localStorage
const persister = createSyncStoragePersister({
  storage: window.localStorage
})

// 2. Envelopper l'app avec le provider
<PersistQueryClientProvider
  client={queryClient}
  persistOptions={{ persister }}
>
  <App />
</PersistQueryClientProvider>

// Résultat : toutes les données (matchs, classements, actus…)
// sont sauvées dans localStorage et rechargées instantanément
Netlify Functions — Proxy Pattern
Netlify Function (proxy)

Petit serveur Node.js qui tourne sur les serveurs de Netlify. Le navigateur appelle /api/v4/matches, la fonction l'intercepte, ajoute la clé API secrète, et retransmet la requête à football-data.org. La clé n'est jamais visible dans le code source.

netlify/functions/football.js
// Le navigateur appelle /api/v4/matches?status=IN_PLAY
// Netlify intercepte → cette fonction tourne côté serveur

exports.handler = async (event) => {
  const apiPath = event.path.replace(/^\/api/, '')
  const qs  = event.rawQuery ? `?${event.rawQuery}` : ''
  const url = `https://api.football-data.org${apiPath}${qs}`

  const res = await fetch(url, {
    headers: { 'X-Auth-Token': process.env.API_KEY }
    // ↑ clé secrète stockée dans Netlify — jamais dans le code
  })

  return {
    statusCode: res.status,   // retransmet le 429 pour que TanStack garde le cache
    body: await res.text(),
  }
}
Netlify Functions — Variables d'environnement
process.env

Les variables d'environnement sont des valeurs secrètes définies dans le tableau de bord Netlify (jamais dans le code). Accessibles uniquement côté serveur via process.env.NOM_VARIABLE. Le navigateur ne les voit jamais.

netlify/functions/
// football.js — clé football-data.org
process.env.API_KEY

// apifootball.js — clé api-football.com (100 req/jour)
process.env.APIFOOTBALL_KEY

// Dans Netlify : Site Settings → Environment Variables → Add variable
// API_KEY = xxxxxxxxxxxxx
// APIFOOTBALL_KEY = xxxxxxxxxxxxx
Pourquoi ? Si la clé était dans le code React (qui tourne dans le navigateur), n'importe qui pourrait l'inspecter avec les DevTools et l'utiliser pour épuiser ton quota à ta place.
Cache — localCache.js
localStorage

Base de données clé-valeur du navigateur. Persiste entre les sessions (contrairement à sessionStorage). Limite ~5MB. Dans ce projet, utilisé pour deux choses : le cache des données API (via localCache.js) et les timestamps des matchs live (via matchStateTracker.js).

src/hooks/localCache.js
const PREFIX = 'foot_'  // toutes les clés du projet commencent par foot_

// Écrire avec TTL (Time To Live)
export function writeCache(key, data, ttlMs) {
  localStorage.setItem(PREFIX + key, JSON.stringify({
    data,
    exp: Date.now() + ttlMs,  // timestamp d'expiration
    savedAt: Date.now()
  }))
}

// Lire — retourne null si expiré
export function readCache(key) {
  const entry = _parse(key)
  if (!entry) return null
  if (Date.now() > entry.exp) {
    localStorage.removeItem(PREFIX + key)
    return null
  }
  return entry.data
}

// Lire même si expiré — fallback si l'API est down
export function readCacheStale(key) { return _parse(key)?.data ?? null }
Cache — TTL adaptatif
TTL (Time To Live)

Durée de vie d'une entrée en cache. Dans ce projet, le TTL s'adapte selon le contexte : si un match est en cours, le cache expire vite pour rester à jour. Si le jour est passé ou futur, le cache dure plus longtemps.

src/hooks/useTodayMatches.js
// TTL différent selon le contexte
const hasLive = matches.some(m =>
  m.status === 'IN_PLAY' || m.status === 'PAUSED'
)
const isToday = dayOffset === 0

const ttl = hasLive
  ? 2 * 60 * 1000      // 2min  — match en cours
  : isToday
    ? 60 * 60 * 1000    // 1h    — aujourd'hui, pas de live
    : 6 * 60 * 60 * 1000 // 6h    — hier ou demain

writeCache(cacheKey, matches, ttl)
Polling intelligent
Polling & Fenêtres temporelles

Au lieu de poller en continu (qui épuiserait le quota de 100 req/jour d'api-football.com), useLiveMinute définit 4 fenêtres critiques par match. En dehors de ces fenêtres, aucune requête n'est envoyée.

src/hooks/useLiveMinute.js
// 4 fenêtres de polling par match :
//   Fenêtre 0 : autour du KO (0-60min depuis utcDate) → détecte le début
//   Fenêtre 1 : à partir de la 45e min → détecte la mi-temps
//   Fenêtre 2 : après 13min de pause → détecte la reprise
//   Fenêtre 3 : à partir de la 90e min → détecte la fin

function isInPollingWindow(match, trackedIds) {
  if (quotaRemaining < 5) return false  // bascule heuristique si quota bas

  const elapsed = (Date.now() - new Date(match.utcDate)) / 60000

  // Fenêtre 0 : KO — s'applique à tous les matchs
  if (elapsed >= 0 && elapsed <= 60 && match.status === 'TIMED') return true

  // Fenêtres 1-3 : seulement les matchs trackés par l'utilisateur
  if (!trackedIds.has(String(match.id))) return false
  // ...
}
Page Visibility API
Page Visibility API

API navigateur qui détecte quand l'onglet passe en arrière-plan ou revient au premier plan. Quand le PC est en veille ou l'onglet caché, Chrome/Safari throttle les intervals et les refetch. Sans ce correctif, les données pourraient être périmées de plusieurs minutes au réveil.

src/hooks/useLiveMatches.js
// Dès que l'onglet redevient visible → refetch immédiat de liveMatches
useEffect(() => {
  const onVisible = () => {
    if (document.visibilityState === 'visible') {
      queryClient.invalidateQueries({ queryKey: ['liveMatches'] })
    }
  }
  document.addEventListener('visibilitychange', onVisible)
  return () => document.removeEventListener('visibilitychange', onVisible)
}, [queryClient])
Fuzzy Matching
Fuzzy matching (correspondance floue)

Technique pour associer deux chaînes similaires mais pas identiques. Ici, utilisé pour associer un match football-data.org à son équivalent ESPN — les deux APIs utilisent des noms d'équipe différents ("Paris SG" vs "Paris Saint-Germain").

src/hooks/useEspnScores.js
// Normalise un nom : minuscules, sans accents, sans espaces superflus
function normalize(name = '') {
  return name.toLowerCase()
    .replace(/[àáâ]/g, 'a')
    .replace(/[éèê]/g, 'e')
    .trim()
}

// Associe un event ESPN à notre match par les 5 premiers caractères
function matchEvent(espnEvent, matches) {
  const espnHome = normalize(homeTeam).slice(0, 5)  // "paris"
  const espnAway = normalize(awayTeam).slice(0, 5)  // "lyon "

  return matches.find(m => {
    const h = normalize(m.homeTeam.name).slice(0, 5)  // "paris"
    const a = normalize(m.awayTeam.name).slice(0, 5)  // "lyon "
    return h === espnHome && a === espnAway
    // 5 chars suffisent pour éviter les faux positifs
  })
}
Promise.all / Promise.allSettled
Promise.all

Lance plusieurs requêtes en parallèle et attend que toutes soient terminées. Si une échoue, tout échoue. Dans ce projet : fetch simultané de IN_PLAY et PAUSED pour le widget live.

src/hooks/useLiveMatches.js
// Les deux fetches partent EN MÊME TEMPS → deux fois plus rapide qu'en séquence
const [r1, r2] = await Promise.all([
  fetch('/api/v4/matches?status=IN_PLAY'),
  fetch('/api/v4/matches?status=PAUSED'),
])
Promise.allSettled

Comme Promise.all mais ne s'arrête pas si une requête échoue. Retourne le statut (fulfilled/rejected) de chacune. Utilisé dans news.js pour que si un flux RSS plante, les 3 autres continuent.

calcMinute() — MT & minutes en local
calcMinute()

Fonction utilitaire qui calcule la minute affichée côté client. Depuis la mi-temps, elle pilote l'affichage en local depuis pausedAt — indépendamment du statut football-data.org (trop imprévisible). Priorité : api-football.com (précis) → estimation pausedAt+15min (fiable) → heuristique utcDate (fallback).

src/utils/matchUtils.js
export function calcMinute(match) {
  if (match.status !== 'IN_PLAY' && match.status !== 'PAUSED') return null

  const state = getMatchState(match.id)

  // MT & 2ème MT — piloté en local depuis pausedAt (indépendant du statut API)
  if (state.pausedAt) {
    // Source précise api-football.com → prioritaire
    if (state.half2Start) {
      const min2 = Math.floor((Date.now() - state.half2Start) / 60_000) + 1
      return min2 <= 45 ? `${45 + min2}'` : `90+${min2 - 45}'`
    }
    // Estimation : MT dure 15min, puis bascule côté client automatiquement
    // Football-data.org peut changer statut trop tôt ou trop tard → on l'ignore
    const sinceP = Date.now() - state.pausedAt
    if (sinceP < 15 * 60_000) return 'MT'
    const min2 = Math.floor((sinceP - 15 * 60_000) / 60_000) + 1
    return min2 <= 45 ? `${45 + min2}'` : `90+${min2 - 45}'`
  }

  if (match.status === 'PAUSED') return 'MT'  // pausedAt pas encore écrit

  // 1ère MT précise via api-football.com (kickoffAt)
  if (state.kickoffAt) {
    const min1 = Math.floor((Date.now() - state.kickoffAt) / 60_000)
    return min1 <= 45 ? `${Math.max(1, min1)}'` : `45+${min1 - 45}'`
  }

  // Heuristique depuis utcDate (aucun timestamp dispo)
  const elapsed = Math.floor((Date.now() - new Date(match.utcDate)) / 60_000)
  if (elapsed <= 45) return `${Math.max(1, elapsed)}'`
  if (elapsed <= 64) return 'MT'
  const half2 = elapsed - 64
  return half2 <= 45 ? `${45 + half2}'` : `90+${half2 - 45}'`
}
matchStateTracker — Timestamps localStorage
matchStateTracker

Module utilitaire qui stocke les timestamps clés d'un match dans localStorage. Permet à calcMinute() d'être précis même après un rechargement de page.

src/utils/matchStateTracker.js
// Clés localStorage pour un match :
// foot_ms_12345 = { kickoffAt, pausedAt, half2Start }

// Détecte les transitions de statut à chaque refetch (football-data.org)
export function trackMatchState(match) {
  const stored = JSON.parse(localStorage.getItem(`foot_ms_${match.id}`) || '{}')

  // Première fois qu'on voit PAUSED → enregistrer pausedAt
  if (match.status === 'PAUSED' && !stored.pausedAt) {
    stored.pausedAt = Date.now()
    localStorage.setItem(`foot_ms_${match.id}`, JSON.stringify(stored))
  }
  // half2Start n'est PAS écrit ici — football-data.org bascule IN_PLAY trop tôt.
  // Seul setHalf2Start() (useLiveMinute / api-football.com) peut l'écrire.
  // calcMinute() gère MT→2ème MT via pausedAt + estimation 15min.
}
